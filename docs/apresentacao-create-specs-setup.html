O problema que o create-specs-setup resolve

Hoje, muitas vezes…

  • A informação fica espalhada: e-mail, wiki, GitLab, conversa
  • Cada pessoa escreve a task de um jeito
  • Ninguém sabe "onde paramos" na feature
  • Quem chega novo perde tempo procurando contexto
  • Documentação fica desatualizada

Depois do create-specs-setup…

  • Tudo fica num lugar só, versionado no Git
  • Modelos padronizados para cada tipo de documento
  • Andamento visível em um arquivo por feature
  • Contexto do produto sempre à mão
  • Um assistente guia passo a passo

Em uma frase

O create-specs-setup configura o nexus-specs (ou outro {projeto}-specs); a partir daí, o time documenta o produto conversando com o assistente no Cursor — sem montar estrutura na mão.

Você não precisa decorar comandos. Diga "Olá" ou descreva o que precisa. O assistente mostra opções e faz as perguntas certas.

O que o create-specs-setup entrega

🧭

Contexto do produto

steering/ — negócio e engenharia. O assistente sempre sabe do que se trata o sistema.

📐

Modelos padronizados

14 templates — meeting notes, regras, use case, tasks de design e dev, etc.

🤖

Roteiros do assistente

17 fluxos guiados no nexus-specs — feature, bug, implementação, status.

Tudo isso nasce na entrevista guiada do create-specs-setup — ver Vídeo 3. Depois, o time usa só o nexus-specs.

Glossário rápido (sem mistério)

create-specs-setup
Ferramenta que cria e configura o repositório de specs de um produto. Usada uma vez no setup.
{projeto}-specs
Repositório gerado — ex.: nexus-specs. Onde o time trabalha no dia a dia.
Cursor
Editor de código com chat de IA. É onde você conversa com o assistente.
Assistente / Agente
O chat de IA. Pense nele como um colega que segue um roteiro e ajuda a escrever docs.
Specs / Especificações
Documentos do produto: o que a feature faz, regras, telas, tarefas para dev e design.
Feature
Uma funcionalidade nova ou mudança grande — tem sua própria pasta com todos os artefatos.
Fix / Ajuste
Correção de bug ou mudança pequena — fluxo mais curto que uma feature.
Revisão humana
A IA gera rascunho; você valida antes de considerar pronto. Nada sai "no automático".

Termos como "skill" ou "agent" existem por baixo dos panos — você não precisa usá-los no dia a dia.

O que existe dentro do repositório

nexus-specs/ ← exemplo real
├── features/ cada feature numa pasta
├── fixes/ bugs e ajustes pontuais
├── steering/ contexto fixo do produto
├── templates/ modelos de documento
└── prototypes/ app Angular mockado (quando houver)
└── src/app/feature/ um protótipo por pasta: 100-listagem-usuario, 101-login…
Na prática, você quase só mexe em features/ e fixes/. O assistente cria e atualiza os arquivos; você revisa e responde às perguntas.

Dois repositórios, dois momentos

🏗️

create-specs-setup

Ferramenta de configuração. Usada na primeira vez (ou para atualizar uma instância antiga). Fica fora do dia a dia.

Analogia: o instalador do create-react-app — não é o app em si.

📋

{projeto}-specs

Repositório do produto. Onde o time trabalha todos os dias. Exemplo real: nexus-specs.

Analogia: o projeto React que o time desenvolve depois.

Depois do setup, o time pode remover create-specs-setup do workspace e usar só o nexus-specs.

Quem faz o setup?

👤 Idealmente

  • Tech lead ou dev sênior
  • Alguém com visão de produto (PO)
  • Quem conhece API, SPA e GitLab do projeto

⏱️ Quanto tempo

  • Uma sessão de entrevista (30–90 min)
  • Respostas em português, no chat
  • Revisão dos rascunhos antes de gravar

🎯 Resultado

  • Repo specs versionado no Git
  • Time inteiro pode começar a usar
  • Mesmo fluxo replicável em outros produtos

PO e designer não precisam rodar o setup — só participam das perguntas de negócio e validam os rascunhos.

Antes de começar — checklist

  1. 1
    Clonar os repositórios create-specs-setup + API + SPA + pasta vazia para o specs (ex.: nexus-specs).
  2. 2
    Abrir tudo no mesmo workspace do Cursor Multi-root: todos os repos visíveis na barra lateral. O assistente precisa "enxergar" API e SPA para validar paths e stack.
  3. 3
    Ter respostas sobre o produto Problema que resolve, quem usa, links do GitLab, se há wiki de regras de negócio, padrão de tasks do time.

Fases 1 e 2 — o coração do setup

Negócio

steering/product.md

  • Qual problema o sistema resolve?
  • Quem são os usuários?
  • O que entrega de valor?
  • O que está dentro e fora do escopo?

O assistente monta um rascunho → você revisa e confirma antes de seguir.

Engenharia

steering/engineering.md

  • Onde ficam API e SPA no workspace?
  • Stack (Java, Angular, etc.)
  • Convenções do time
  • O assistente lê os repos — não inventa

Esse arquivo orienta dev e IA em todas as specs futuras.

Decisões importantes na entrevista

📚 Wiki de regras de negócio

O time publica RNs numa wiki do GitLab (ex.: nexus-spa)? Se sim, o assistente configura links e numeração padrão (RN-{code}.{seq}).

📝 Template de tasks

O time já tem formato de issue no GitLab? Pode manter o de vocês ou usar o template padrão do setup. O assistente adapta os modelos.

🚨 Hotfix urgente

Correção crítica em produção vai direto no GitLab, sem passar pelo fluxo de specs. O setup registra essa regra.

🖥️ Protótipo Angular

Opcional. O assistente copia componentes visuais do SPA real e monta um app navegável mockado — útil para validar telas com PO e designer.

Regra de ouro do setup

Nenhum arquivo é criado até você ver o resumo final e dizer "sim" explicitamente.

O que nasce depois do "sim"

Arquivos criados automaticamente

  • sdd.config.yaml — mapa do projeto
  • steering/ — contexto de negócio e engenharia
  • templates/ — 14 modelos de documento
  • .agents/skills/ — roteiros do assistente (17 no Nexus)
  • features/ e fixes/ — pastas vazias, prontas
  • prototypes/ — se confirmado na entrevista

Validação antes do handoff

  • Checklist automático (V1–V11)
  • Nenhum placeholder {{…}} solto
  • Tabela ✅ / ❌ apresentada a você
  • Lista de pendências não críticas

Só encerra quando tudo estiver consistente.

Handoff — liberando o time

Regras simples para todo mundo

✅ Faça

  • Responda as perguntas do assistente com o que você sabe
  • Revise o rascunho antes de seguir para a próxima etapa
  • Diga "não sei" ou "pendente" — o assistente marca

⛔ Evite

  • Assumir que o rascunho está 100% certo sem ler

  • Documentar hotfix urgente aqui — vai direto no GitLab
  • Ter medo de errar — tudo fica no Git e pode ser corrigido

Grave cenários separados: feature nova · bug · consulta de status

Três fluxos que você vai usar

✨

Feature nova

Do zero até tasks no GitLab. Mais longo, mais completo. PO e requisitos lideram o início; design e dev entram depois.

Bug ou ajuste

Fluxo curto. Descreva o problema → task de correção → dev implementa. Ideal para QA e dev.

📷

Já existe no sistema

Tela ou módulo já implementado, falta documentar. Dev ou requisitos pedem; assistente lê o código e gera a spec.
