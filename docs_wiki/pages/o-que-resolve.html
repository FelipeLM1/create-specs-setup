O que o Create Specs Setup resolve

Times misturam decisão de produto em e-mail, wiki, issue e conversa — e na hora de implementar ninguém sabe o que ficou acordado.

O problema

Especificar software no dia a dia costuma ser fragmentado: a conversa some, a regra de negócio fica só na cabeça de alguém, e a task no GitLab não conta a história completa.

Quando o time cresce ou o projeto já está em produção, falta um lugar único — com linguagem de produto — onde feature, ajuste e status vivam juntos.

O que o sistema resolve

O Create Specs Setup não é onde você especifica features todos os dias. Ele é o bootstrap: uma entrevista guiada que monta o repositório {projeto}-specs completo — prontinho para o time trabalhar.

Analogia

Como o create-react-app para apps: você configura uma vez; depois vive no projeto gerado.

Na prática, o valor aparece em três frentes:

  • Alinhamento — negócio e engenharia descritos no steering desde o primeiro dia
  • Ritmo — skills e templates prontos para feature, bug, spec a partir do código, implementação guiada
  • Continuidade — upgrade de instâncias antigas sem apagar o que já existe

Como funciona (visão de produto)

Do bootstrap ao dia a dia

  1. 1

    Conversar

    Você abre o chat no create-specs-setup.

  2. 2

    Entrevistar

    O assistente pergunta e monta o resumo.

  3. 3

    Confirmar

    Só grava depois do seu “sim”.

  4. 4

    Viver no specs

    O dia a dia muda para {projeto}-specs.

  1. Você abre o chat no repositório create-specs-setup e diz o que precisa (setup novo ou atualização).
  2. O assistente entrevista em etapas — produto, engenharia, wiki de RN, GitLab, protótipo, docs wiki — e só grava depois do seu “sim”.
  3. Nasce o {projeto}-specs com pastas, skills e configuração do seu time.
  4. O dia a dia muda de casa: feature, bug e status acontecem no repositório de specs, não no bootstrap.

Para quem é

Para times que querem especificação guiada por conversa (PO, dev, líder), com artefatos versionados e um contrato claro do que a instância SDD precisa ter.

Fora de escopo aqui

Esta página não ensina a implementar código nem lista endpoints. O foco é o papel do produto no processo do time.
