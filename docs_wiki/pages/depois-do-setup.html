Onde trabalhar

Abra o chat no repositório {projeto}-specs e diga o que precisa. O assistente de boas-vindas mostra as opções e encaminha o fluxo certo — sem exigir que você saiba o nome de cada skill.

Dois repositórios, papéis diferentes

create-specs-setup

Configurar ou atualizar a instância. Usado poucas vezes.

vs

{projeto}-specs

Feature, bug, status, implementação e docs wiki. Casa do dia a dia.

O que você pode pedir

  • “Quero especificar uma feature nova”
  • “Documente o fluxo que já está na tela”
  • “Tenho um bug na listagem”
  • “Onde paramos na feature X?”
  • “Implemente a tarefa Y” / “Investigue o bug Z”
  • “Gere uma página da wiki sobre…”

Principais fluxos (em linguagem de produto)

  1. Feature nova — da conversa/reunião até regras, casos de uso, tasks e (se quiser) protótipo.
  2. Documentar o que já existe — ler o sistema atual e gerar a spec a partir do comportamento real.
  3. Bug ou ajuste — registro enxuto do problema e do que deve mudar.
  4. Implementar — o dev avança com plano confirmado, alinhado à spec e aos padrões do projeto.

Hotfix urgente

Produção quebrada não passa pela pasta de fixes do specs: vai direto no GitLab com MR, conforme o fluxo do time.

Docs wiki ao longo do tempo

Se a docs wiki foi habilitada no setup (como esta que você está lendo), novas páginas nascem sob demanda: peça para documentar um tema ou uma feature em tom de produto.

Lembrete

A wiki de produto é diferente da wiki de regras de negócio no GitLab (quando o time usa). Uma conta a história; a outra numera RNs para as tasks.
